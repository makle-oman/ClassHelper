# ClassHelper - 前端静态页面需求文档

## 1. 文档目标

本文档用于约束教师端前端静态页面阶段的功能范围，不考虑后端接口、数据库和真实鉴权，只关注：

- 页面结构是否完整
- 页面跳转是否通畅
- 本地 Mock 数据下的交互是否闭环
- 表单、弹窗、筛选、导入提示、空状态等静态功能是否齐备
- UI 是否达到可继续对标接口开发的程度

完成本文档后，前端页面应尽量做到“界面完整、流程清晰、字段明确、交互闭环”，后续接口开发只需要按文档对接数据。

---

## 2. 当前阶段范围

### 2.1 本阶段包含

- 登录 / 注册静态页面
- 教师端首页与 Tab 导航
- 班级、学生、学期、课程表、考勤、成绩、作业、通知、请假审批、个人中心、设置等页面
- 本地数据驱动的新增、编辑、删除、筛选、切换、导入提示、统计展示
- 前端字段与交互规则梳理
- 前端 UI 一致性优化

### 2.2 本阶段不包含

- 真实接口对接
- 真实登录鉴权
- MySQL / TypeORM / NestJS API
- 文件真实上传下载
- 推送服务端能力

## 4. 静态页面完成标准

每个页面在静态阶段需要满足以下标准：

- 可以从合理入口进入
- 页面主操作按钮明确
- 表单字段完整，文案与需求一致
- 有成功、取消、删除确认等基本反馈
- 有空状态、无结果状态或占位提示
- 有本地 Mock 数据支撑主要流程
- 不出现明显断链、死按钮或无反馈控件

## 5. 模块需求清单

以下状态说明：

- `已具备`：页面和核心交互已存在
- `部分具备`：页面已存在，但仍有关键静态能力待补
- `待补充`：需求明确，但前端还未完成或完成度不足

### 5.1 登录与注册

状态：`部分具备`

#### 需求

- 登录页支持手机号、密码输入
- 登录按钮可进入主页，作为静态流程替代真实鉴权
- 注册页支持姓名、手机号、密码、确认密码
- 登录页与注册页可互相跳转
- 根入口页需要明确静态跳转策略

#### 待补充

- 启动页 / 根入口需要补一个明确的登录态判断占位逻辑说明
- 登录与注册的前端校验规则需要统一整理到文档

### 5.2 首页

状态：`已具备`

#### 需求

- 展示教师问候、头像、班级数、学生数、今日课程、出勤率
- 提供快捷入口：学生、考勤、作业、通知、成绩、课程表
- 展示今日课程卡片
- 展示待办提醒
- 点击卡片和快捷入口可跳转对应页面

#### 优化建议

- 首页统计与待办内容后续应抽象成统一卡片数据结构
- 待办提醒建议补一个“已处理/忽略”静态交互占位

### 5.3 学生管理

状态：`已具备`

#### 需求

- 按班级切换学生列表
- 按姓名 / 学号搜索
- 支持手动新增学生
- 新增学生使用表单弹窗，字段至少包括：
  - 姓名
  - 学号
  - 性别
  - 班级
  - 家长姓名
  - 家长电话
- 支持 Excel 导入提示
- 导入弹窗中需要提供模板下载提示
- 学生列表点击进入学生详情
- 空列表与无搜索结果时给出明确提示

#### 待补充

- 静态阶段可补一个“导入字段说明”或“模板字段预览”弹窗
- 可补一个“删除学生入口从详情页返回列表联动”的文档说明

### 5.4 学生详情

#### 待补充

## 6. 前端静态阶段优先级建议

### P0：先补齐，作为静态页面可交付底线

- 考勤统计页补“按学生查看出勤率”
- 课程表补“今日快捷定位”
- 根入口补明确的静态登录态跳转说明
- 所有导入能力补统一模板说明文案
- 梳理主要表单的必填字段和校验文案

### P1：增强页面闭环，提升后续接口对接清晰度

- 学生、成绩、课程表导入字段说明统一
- 作业、通知、请假页补筛选和状态说明
- 学期与课程表增加静态联动说明
- 关键页面补空状态、无数据状态、无搜索结果状态

### P2：体验打磨

- 统一导航栏、标题、浮动按钮样式
- 统一筛选区、搜索区、卡片间距和弹窗表单结构
- 优化首页信息层级和可点击性

## 7. UI 优化建议

以下建议不影响静态功能闭环，但建议在前端阶段同步处理。

### 7.1 一致性优化

- 顶部导航样式目前存在两套：Tab 页无统一标题区，业务页多为自定义导航栏，建议统一标题、返回按钮热区、右侧操作位宽度
- 搜索栏、筛选 Chip、浮动按钮、弹窗底部按钮的圆角和高度建议统一
- 列表卡片中阴影、边距、内边距有轻微差异，建议统一成 2 到 3 套标准卡片规格

### 7.2 信息层级优化

- 首页信息密度较高，但“统计、快捷入口、今日课程、待办”之间层级还可以更鲜明
- 课程表页功能最复杂，顶部操作较多，建议后续增加一级功能分组，避免按钮堆叠感
- 成绩页和考勤页的统计视图可以加强筛选标题区，让用户更容易理解当前统计口径

### 7.3 交互反馈优化

- 目前很多操作成功反馈依赖 `Alert`，静态阶段可继续用，但后续建议统一为轻提示或底部 Toast
- 删除、归档、升迁、审批这类有后果的动作，可以增加更明确的二次确认文案
- 导入相关交互建议统一为三段式：说明 -> 模板 -> 选择文件 / 预览

### 7.4 视觉优化

- 当前整体风格稳定，主色绿色方向适合教师工具
- `scores`、`notices` 等页面的色彩标签较丰富，但不同模块的强调色规则还不完全统一，建议统一“状态色”和“业务色”
- 局部页面存在信息卡片过多、每张卡信息稍少的问题，可适当减少重复卡片

### 7.5 可用性优化

- 课程表、成绩录入、学生列表都属于高频页，建议优先优化“少点一步”的操作
- 课程表应尽快补“今日快捷定位”
- 考勤统计应尽快补“按学生查看”
- 学生和成绩导入都建议补模板字段预览，减少用户理解成本

---

## 8. 静态页面阶段完成标志

满足以下条件后，可视为教师端前端静态需求基本完成：

- 核心业务页全部可进入、可返回、可完成主要静态操作
- 所有页面字段和交互已明确，不再频繁改动结构
- 前端文档可以直接指导接口字段设计
- 页面中不再存在明显“只有壳没有交互”的核心模块
- 只剩数据联调、状态管理、接口异常处理等动态问题

---

## 9. 后续执行建议

建议后续按以下顺序推进前端静态完善：

1. 先补 `attendance` 的学生出勤率统计切换
2. 再补 `schedule` 的今日快捷定位
3. 统一导入说明、模板文案和字段说明
4. 统一表单校验与空状态文案
5. 最后集中做一轮 UI 一致性优化
